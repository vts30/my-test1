SUMMARY
-------
Between 14:10 and 14:30 on 18.03.2026, the forumsuite
application experienced a severe JVM memory and garbage
collection (GC) event. The JVM allocated 131 GB of objects
within approximately 2 hours, resulting in application thread
suspensions, major GC cycles, and potential user-facing
slowdowns or timeouts.

TIMELINE
--------
13:40       Normal baseline operation
~14:00      GC activity begins increasing (minor GC spikes)
~14:10      First major allocation spike: ~15 GB in short burst
            Eden Space exhausted, promotion to Old Gen begins
~14:15      JVM thread suspension spike (~10%)
            All application threads paused (Stop-The-World GC)
~14:20      Second allocation spike: ~10 GB
            Old Gen usage peaks at ~10 GiB
            GC pause time reaches ~60ms
~14:30      Old Gen GC completes, memory partially released
14:30+      Eden Space GC frequency increasing (not stabilized)

IMPACT
------
- User-facing: Likely slowdowns or request timeouts
  during 14:10–14:30
- System: Excessive CPU consumed by GC during the event
- Risk: OutOfMemoryError possible if load increases
  before fix is applied

ROOT CAUSE (Preliminary)
-------------------------
Dynatrace flame graph and allocation hotspot analysis
identified the following:

1. JSONParser — High allocation on every HTTP request.
   JSON deserialization objects are likely being created
   per-request instead of using shared/singleton instances.

2. HashMap — Excessive HashMap creation in the request
   processing path contributing to GC pressure.

3. Spring Filter Chain — Multiple OncePerRequestFilter
   implementations building large objects per request
   (CharacterEncodingFilter, ServerHttpsObservationFilter,
   FormContentFilter, etc.)

The combination of these allocation patterns overwhelmed
the G1 garbage collector, causing Eden Space to fill up
faster than it could be collected, promoting large amounts
of short-lived objects into Old Gen, and triggering
expensive Major GC cycles.

BACKGROUND
----------
A production performance incident occurred on 18.03.2026
between 14:10–14:30. Dynatrace recorded 131 GB of memory
allocated by the forumsuite JVM in ~2 hours, causing
Stop-The-World GC events, thread suspensions, and potential
user-facing timeouts.

Dynatrace flame graph and allocation hotspot analysis
identified the root cause in the HTTP request processing path.

PROBLEM
-------
The following are confirmed allocation hotspots visible
in the Dynatrace flame graph (allocation path):

  Thread
  └── ThreadPoolExecutor (Tomcat worker threads)
      └── NioEndpoint → Http11Processor → CoyoteAdapter
          └── Spring ApplicationFilterChain
              └── OncePerRequestFilter (multiple)
                  └── [Application code]
                      ├── JSONParser       ← HOTSPOT
                      └── HashMap          ← HOTSPOT

1. JSONParser allocations on every request
   - Likely cause: ObjectMapper or JSON parser instance
     created per-request instead of being shared/singleton
   - Fix: Inject ObjectMapper as a Spring @Bean (singleton)
     and reuse it across all requests

2. HashMap over-allocation
   - Large or numerous HashMaps created per request
   - Fix: Audit request-scoped code that builds Maps;
     reuse, cache, or replace with lighter structures
     where appropriate

3. Spring Filter Chain object creation
   - Multiple filter implementations creating objects
     on every request
   - Fix: Review each custom OncePerRequestFilter for
     per-request allocations that could be cached
     or reduced
